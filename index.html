State Machine Module

ROS node that allows to move across the given topological map performing the task as stated in the project requirements. In the following, the behavior of the robot is implemented as a Smach finite state machine. For this purpose, each state is associated with a class that includes an execute function, that implements the actual logic corresponding to that state.

class scripts.state_machine.Move_to_recharge(interface_helper, ontology_manager, env_size)[source]

Class associated to the MOVE_TO_RECHARGE state in the smach finite state machine. In this state the robot keeps moving randomly on corridors only until it can reach the recharging base.

Note

Since each room with at least two doors is considered as a corridor, only the peripheral rooms are not considered such. This fact allows to consider valid this approach and guarantees, without any further assumption on the topological map, that after a large enough amount of time, the target room will be reached.

execute(userdata)[source]

Core function for the REASONING state. After retrieving the reachable rooms at this step, if, among them there is one that is urgent (meaning that the time elapsed from the last visit of the robot is above the threshold) the robot goes there. However, if the urgent rooms are more than one, the logic picks a random one among them. In case of no urgent rooms, the robot prefers to stay on corridors, until a room becomes urgent or battery level gets too low. In case of more corridors available, the robot gives priority to urgent ones and, if noone of them is such, it chooses randomly one to go to.

Parameters

userdata (Remapper) – structure that stores data exchanged between states, here used to propagate the target room.

Raises

rospy.ROSException – if no room is reachable starting from the current one.

Returns

One among the possible following states in the finite state machine (PLANNING, MOVE_TO_RECHARGE).

Return type

str

scripts.state_machine.main()[source]

Main function for ROS node definition and smach finite state machine execution. In this method all states are connected with transitions.

Subscribes to:

  • state/battery_low: where a given node updates robot’s battery status (through the instanciated InterfaceHelper object).

Services:

  • state/set_pose (client): client to setting initial pose of the robot (through the instanciated InterfaceHelper object).

  • /armor_interface_srv (client): used to perform calls to the ARMOR ROS service (through the instanciated OntologyManager object)

Action servers:

  • motion/planner (client): client to ask the planner for the list of waypoints (through the instanciated InterfaceHelper object).

  • motion/controller (client): client to execute the plan and move the actuators (through the instanciated InterfaceHelper object).
